# abyo-speculate Benchmarks

Honest, reproducible numbers. We do not quote a single "X× faster" headline —
SD speedup depends heavily on the target/draft pair, the task, and the chosen
`draft_lookahead`.

## Hardware

| Component | Spec |
|-----------|------|
| GPU | NVIDIA GeForce RTX 4070 Ti SUPER (16380 MiB, compute 8.9 / Ada Lovelace) |
| Driver | 595.58.03 |
| CUDA toolkit | 12.4.131 |
| candle | 0.8.4 |
| dtype | BF16 (`--features cuda`) |

## Reproducing

Each row was generated by:

```sh
cargo run --release --features cuda --bin abyo-speculate-bench -- \
    --target <TARGET> --draft <DRAFT> --family <FAMILY> \
    --task <TASK> --method both \
    --max-tokens 128 --warmup 1 --runs <N> --draft-lookahead <K>
```

Models are pulled from Hugging Face on first run and cached at
`~/.cache/huggingface/hub`. `--family auto` (default) infers from the repo id.

## Per-task: Qwen 2.5 3B + 0.5B draft, k=4 (vanilla SD, BF16)

128 new tokens, temperature 0.7, top_p 0.95, 1 warmup + 3 runs each.
Re-measured at v0.3.0 (numbers below are the **mean of 3 runs**).

| Task | AR tok/s | SD tok/s | Speedup |
|------|---------:|---------:|--------:|
| chat | 34.4 | 49.5 | **1.44×** |
| **code** | **35.3** | **61.4** | **1.74×** |
| translation | 34.1 | 48.0 | **1.41×** |
| long_context | 31.5 | 49.6 | **1.57×** |

`code` wins by a wide margin — coding tokens are highly predictable
(whitespace, brackets, common keywords), so the draft model's acceptance
rate is highest. `translation` and `chat` cluster around 1.4×; `long_context`
lands in between, with autoregressive baseline slightly slower (KV cache
reads dominate).

## Per-pair: target × draft (default chat task, k=4)

128 new tokens, 3 measured runs after 1 warmup.

| Target | Draft | AR tok/s | SD tok/s | Speedup | Notes |
|--------|-------|---------:|---------:|--------:|-------|
| Qwen 2.5 1.5B | Qwen 2.5 0.5B | 60.4 | 59.7 | 0.99× | Param ratio (3×) too small to win |
| **Qwen 2.5 3B** | **Qwen 2.5 0.5B** | **34.3** | **49.0** | **1.43×** | Sweet spot for this GPU |
| Qwen 2.5 7B | Qwen 2.5 0.5B | 18.5 | OOM | N/A | Both models don't fit in 16 GB |

The 1.5B/0.5B pair shows the lower-bound: when target ≈ draft, SD's
per-round overhead eats the parallelisation win. The 7B/0.5B pair would
likely show the biggest speedup but doesn't fit alongside the draft on a
16 GB card without quantisation.

## `draft_lookahead` sweep on Qwen 2.5 3B + 0.5B

Same 128-token / 3-run setup, varying `k` only:

| k | AR tok/s | SD tok/s | Speedup |
|---|---------:|---------:|--------:|
| 2 | 35.2 | 47.8 | 1.36× |
| 3 | 35.1 | 47.7 | 1.36× |
| **4** | **34.3** | **49.0** | **1.43×** |
| 6 | 34.4 | 31.8 | 0.92× |
| 8 | 30.5 | 28.8 | 0.95× |

`k = 4` is optimal for this pair — beyond that, the per-round acceptance
rate drops fast enough that wasted draft work outweighs the
parallelisation. This matches the qualitative analysis in Leviathan et al.
2023 §3.4.

## Medusa: loader + plumbing verified, speedup pending 24 GB+ hardware

Two integration tests cover the Medusa story end-to-end:

1. **`tests/with_real_medusa_heads.rs`** — Heads-only loader. Parses
   `FasterDecoding/medusa-vicuna-7b-v1.3`'s `medusa_lm_head.pt` (5 heads,
   4096 hidden, 32000 vocab, BF16) via
   `MedusaHeadsCandle::from_fasterdecoding_pt`, runs forward over a
   synthetic hidden state, asserts non-NaN logits + in-vocab top-k.
   **Passes on 16 GB.**

2. **`tests/with_real_medusa_e2e.rs`** — Full Vicuna 7B base + bundled
   heads via `MultiPthBackend` (a custom `SimpleBackend` that loads
   PyTorch sharded `.bin` directly — no offline conversion needed).
   The base model loads fine; OOM hits when the 5 Medusa heads layer
   on top, because Vicuna 7B BF16 (~14 GB) + 5 heads (~1.5 GB) +
   activations exceeds 16 GB. **Gated behind `ABYO_LARGE_GPU=1`** so
   it skips on commodity 16 GB hardware. Expected to pass on 24 GB+
   cards (RTX 4090 / RTX 5090 / A10G / L4).

Speedup numbers will land in this section once we measure on a 24 GB+
card. The plumbing is verified; only the headline number is pending.

## EAGLE-2 / EAGLE-3 (v0.3.0)

### EAGLE on Q4 targets — honest negative result

`tests/with_eagle_e2e.rs` and `tests/with_eagle3_e2e.rs` cover Llama 3
8B Q4_K_M + EAGLE-LLaMA3 / EAGLE3-LLaMA3.1 respectively. After the
v0.2.2 `tree_logits` fix (single-position GEMV vs multi-position GEMM
precision drift), both produce **byte-for-byte the AR baseline output**
under greedy acceptance — i.e. correctness is intact.

| Method (target / draft, depth=4 k=2 dyn=16, 32 tokens) | AR tok/s | SD tok/s | Speedup |
|--------------------------------------------------------|---------:|---------:|--------:|
| EAGLE-2 Cartesian, Llama 3 8B Q4_K_M + EAGLE-LLaMA3-8B | 45.0 | 6.6 | **0.15×** |
| EAGLE-2 Dyn-16,    Llama 3 8B Q4_K_M + EAGLE-LLaMA3-8B | 45.0 | 8.5 | **0.19×** |
| EAGLE-3 Dyn-16,    Llama 3.1 8B Q4_K_M + EAGLE3-LLaMA3.1-8B | 45.8 | 9.5 | **0.21×** |

The SD numbers are **lower than AR** here. Two compounding effects:

1. **EAGLE drafts are trained on FP16 hidden states.** Feeding Q4
   target hiddens introduces enough quantization noise that draft
   predictions drift from target's argmax → low acceptance →
   per-token cost approaches per-round cost.
2. **Q4 lm_head dominates per-call cost on this GPU.** A Q4 × 128k
   vocab QMatMul is ~50 ms; AR runs that exactly once per token.
   EAGLE-2's draft loop runs it `draft_depth` extra times per round.

We ship these tests so users can reproduce the negative result and
understand the Q4 trade-off.

### EAGLE on BF16 targets — `tests/with_eagle_bf16_e2e.rs`

`NousResearch/Llama-2-7b-chat-hf` (13.5 GB safetensors mirror of
`meta-llama/Llama-2-7b-chat-hf`, no gating) +
`yuhuili/EAGLE-llama2-chat-7B`. Total GPU footprint: ~15 GB → fits a
16 GB card.

3 prompts × 64 tokens, AR forced to same N (no EOS shortcut), EAGLE-2
depth=2 k=2 (no dynamic prune):

| Run | AR tok/s | EAGLE tok/s | Speedup |
|-----|---------:|------------:|--------:|
| Mean across 3 prompts | 21.2 | 10.3 | **0.49×** |

Greedy acceptance is correct for the first ~20-30 tokens (output
matches AR byte-for-byte). After that the trajectories diverge — the
suspected cause is multi-position GEMM precision drift on
`per_node_logits[i > 0]` cascading once a path commits more than one
draft token (v0.2.2's root-replacement fix only patches index 0). Worth
investigating in v0.3.x for the deeper-tree case.

Llama 2 7B is MHA — KV reads are cheap, so AR per-step is fast (~45 ms)
relative to EAGLE's per-round overhead (~150 ms at depth=2 k=2 + commit
cost). On GQA architectures (Llama 3 8B BF16 would be ideal) the
expected math says EAGLE should be 1.5-2× — but Llama 3 8B BF16
(~16 GB) doesn't leave room for the EAGLE checkpoint on a 16 GB GPU.

## What's NOT in this table (yet)

- **Real Medusa speedup numbers** — see above (24 GB+ GPU needed).

- **Multi-batch serving.** vLLM territory; out of scope.

## Honesty notes

- These numbers are **single-user, batch size 1**. Multi-batch SD is a
  different regime where vLLM / SGLang win and abyo-speculate is not the
  right tool.
- `tok/s` excludes prompt processing. The bench warmup absorbs first-token
  latency.
- BF16 on Ada is a near-no-op accuracy hit vs F32. On Pascal / Volta it
  would degrade more.
- We do **not** assert that SD output is bit-identical to AR output —
  rejection sampling produces samples from the same target distribution,
  but trajectory-level you'd need temperature 0 + matching seeds.
